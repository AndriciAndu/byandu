Client-Side Routing

Functionality : Route (Client Side)

Working Prototype of a Client-Side Routing System, for simulating SPA / AJAX-based functionality.

* Updates parts of an application (the View), without triggering a page reload, simulating a Single-PAge Application functionality

* No Hash Routing required ( None of this : myDomain/#/myPage.html )

* Supports direct navigating to a subsequent Page, without server-side Routing ( ex : mySite/something/etc.html )

* Customizable Intermediary Functions, to control what happens during each Page load.

* Includes a certain degree of automatic Memory Management

- automatic de-refferencing of discarded HTML Elements

- automatic removal of Asterisk Components' Event Listeners

More efficient Page Load Time than traditional Multi-Page Applications, by having most CSS/JS already parsed before each Page Load.

* Slower than traditional SPAs, as individual Page HTML/JS files are still reuested from the server.

Faster 'First-Load' Time as only the used files are downloaded.

* Slower than traditional SPAs, when caching is involved.

* currently supports only static paths

Within a Tradition (multi-page) Website/Application, each individual Page is 'self-sustained', having it's own DOCTYPE Declaration, Meta Tags, CSS/JS Files, HTML markup, etc.

When navigating to different pages, the browser discards the current page completely, requests new files from the server, and loads a completely new Document.

Within a Single-Page Application (SPA) environment, there is a Root Page which is 'self-sustained', just like any Page within a Tradition (multi-page) Website/App.

* The Root Page is ussualy the index.html Page.

Navigating to different pages happens by updating part of the Root Page (the part that is updated is called a View), and this happens without actually navigating away from the Application's Root Page * Hence the name of 'single-page'

Traditional Single-Page Applications

Normally, Single-Page Applications download all the files/data required for the entire application to run, at the very start.

From there on, loading new Pages happens extremely fast, as it removes the need to make requests to the server to get the required files (as the files are already downloaded).

This results to a bigger initial load time, as a lot of data is requested from the server.

Yet, usually, this initial load time is managed by caching the files to the user's device. Doing so, while the very first time the user accesses the application, the initial load time will be longer (getting all the files/data), from there on the initial load time will be short (as the files are cached).

The asterisk-route approach

With the asterisk route things are different. It mimics the functionality of a Single-Page Application, but it functions as a hybrid between a SPA and a multi-page application.

All common CSS/JS Files (utility / components / effects / etc) are refferenced within the Root Page. When the Root Page loads, these common files are available for all 'individual' Pages that load further on - resulting in a faster load than a multi-page application.

Yet, it does make requests to the server to fetch each 'individual' page's .html and/or .js files, when a new page is loaded. This allows for the initial load time to be faster than a SPA approach, when caching is not used, although it has a slower load time for 'individual' Pages, than with a traditional SPA approach.

* The Root Page is basically the index.html Page. It is created like any (multi-page) website's Page, and has the structure of a generic stand-alone page.

The users will never navigate away from the Root Page - when a new Page loads, it will simply trigger an Url change, and the content of the View Element (which is within it) will be updated.

Because of that, we need to add all the website's details (meta tags, titles, CSS/JS file refferences, etc) on the Root Page. They will also be available, by default, for every new Page that is loaded within the View.

Step 1. Create a normal, stand-alone index.html File.

Step 2. Within the Root Page, add the <div id="__route__mainView"></div> (representing the View - where each individual Page's content will be loaded)

Step 3. At the bottom of the Root Page, just before the closing </body> Tag, add the asterisk-route.js JS File. 


															    // Root Page Example 

															    <!DOCTYPE html>
															    <html>
															      <head>
															          // all the application's Meta Tags 
															          // all general-purpose CSS files 
															      </head>
															      <body>
															          // other HTML content 
															          <div id="__route__mainView"> </div>    // this is the Main View, where each Page's content is loaded 
															          // other HTML content 
															          // all general-purpose JS files
															          <script src="--myPath--/asterisk-route.js"></script>
															      </body>
															    </html>

Each 'individual' Page consists of a .html File, the content of which will be placed within the View, whenever that Page is loaded.

Step 1. Create the myPage.html HTML File.

Step 2. Refference the asterisk-route-intermediary.js JS File, at the top of markup.

Step 3. Create an entry within the asterisk.route.paths 


															    // 'Individual' Page Example 

															    <script src="--myPath--/route_intermediary.js"></script> // * Required 
															    <style> /* Add page-specific styles */ </style>          // * Optional
															    <article>
															        // Place content here 
															    </article>

* Each individual Page does not require refferencing to the CSS/JS Files which are already loaded on the Root Page.

* Specific styles can be loaded for the individual Page, within the <style></style> Tag at the top of the HTML File.

* Specific scripts can be loaded for each individual Page, by creating a myArticle.js File and refferencing it within the asterisk.route.paths.

When using Server-side Routing, when a user follows a specific link (ex: http://mySite.com/something.html), it is easy to redirect to the Root Page (ex: http://mySite.com), by default.

With Client-side Routing, when a user follows a specific link (ex: http://mySite.com/something.html), the specific Page will load - this is a problem as 'individual' Pages have only the content which should update the View of the Root Page, and not the Root Page's markup, CSS/JS refferences and so on.

1. The asterisk-route-intermediary.js file contains the code to simply create a custom Url, which contains the details about the Page which has loaded.

http://mySite.com/something.html // Initial Url
http://mySite.com/?load=/something.html // Modified Url

2. With that custom Url, it redirects to the Root Page.

3. When the Root Page is loaded, it checks the url parameters we defined, and loads the content of the Page on which the user initially loaded, within the View.

Refferencing the asterisk-route-intermediary.js JS File within each Page's HTML allows for :

1. It will only execute if the user follows a specific link (ex: http://mySite.com/something.html), where the Root Page is not loaded yet, without parsing the rest of the Page's HTML markup or refferenced Files.

2. It will not execute if the Root Page is already loaded - as adding a <script> through innerHTML will not execute it.

As we never navigate away from the Root Page (even though the url changes), to load a new 'individual' Page within the View, we need to define what files will be used in this process.

For this, we must create a Path Object for each 'individual' Page within our application. These are stored within the asterisk.route.paths Array.

Each Path Object is an unnamed Object, with 3 Properties :

1. filePath_html
Required
String
The path to the Page's .html File
2. filePath_js
Optional
String
The path to the Page's .js File
3. filePath_html
Optional
Object
Stores any parameters required to bring additional changes to Page / Application
Property Mandatory Type Description
filePath_html Yes String The path to the Page's .html File
filePath_js No String / Null The path to the Page's .js File
articleParameters No Object Stores any parameters required to bring additional changes to Page / Application

Within this website (byandu.netlify.com), each Path Object has a category property, which is used to update the Category Buttons within the left-side navigation menu. 


														    asterisk.route.paths = [
														        { 
														            filePath_html : '/myCategory/myArticle_1.html' , 
														            filePath_js   : '/js/myCategory/myArticle_1.js' , 
														            articleParameters : {
														                category : 'myCategory'
														            }
														        } ,

														        { 
														            filePath_html : '/myOtherCategory/myArticle_2.html' , 
														            filePath_js   : null , 
														            articleParameters : {
														                category : 'myOtherCategory'
														            }
														        } 
														    ];

There are 4 Intermediary Functions that can be used to control what happens when a new Page is loaded. They are optional.

These are already defined as empty functions, within the asterisk-route.js JS File, to avoid Refference Errors.

This can be done within the asterisk-route.js File, or within a separate .js File - as long as the separate .js File comes after the asterisk-route.js File.

These functions can be found at the bottom of the asterisk-route.js File. 


															    <script type="text/javascript" src="--myPath--/asterisk-route.js"></script>
															    <script type="text/javascript" src="--myPath--/mySeparateFile.js"></script>

1. asterisk.route.intermediary.loadPage_onBegin()

This runs immediately when a new Page starts loading. It can be used to add a loading animation on the View, while the content is updating.

Example : In the current website (byandu.netlify.com) it is used to add a faSpinner loading animation, to the View. 


																		    asterisk.route.intermediary.loadPage_onBegin = function() {
																		        asterisk.route.info.mainView.classList.add('faSpinner', 'faSpinner-fast', 'faSpinner-lg');
																		    };

2. asterisk.route.intermediary.loadPage_updateDOM()

This is used to bring changes to the Elements which are outside the View, when loading a new Page.

Example : In the current website (byandu.netlify.com) it is used to update the active status on the categories/articles buttons within the left side navbar.

* In the example, the articleParams__obj reffers to the current Path Object, defined within asterisk.route.paths. 


																		    asterisk.route.intermediary.updateDOM = function(articleParams__obj) {

																		        var myTargets_1 = Array.from(document.getElementsByClassName('__route__category'));
																		        var identifierValue = articleParams__obj.articleParameters.category;

																		        myTargets_1.map(function(elem) {
																		            if (elem.getAttribute('data-categoryName') == identifierValue) { 
																		                elem.classList.add('active') 
																		            } else { 
																		                elem.classList.remove('active') 
																		            }
																		        });

																		        var myTargets_2 = Array.from(document.getElementsByClassName('__route__pageLink'));
																		        var identifierValue = articleParams__obj.filePath_html;

																		        myTargets_2.map(function(elem) {
																		            if (elem.pathname == identifierValue) { 
																		                elem.classList.add   ('active') 
																		            } else { 
																		                elem.classList.remove('active') 
																		            }
																		        });
																		    };

3. asterisk.route.intermediary.loadPage_runDefaultScript()

This is used as a default script, for the content within the View, after a new Page has loaded. Checks/functions which should occur for every new Page load, should be included here to avoid repeating them across each Page's .js File.

Example : In the current website (byandu.netlify.com) it is used to do some formatting on the text within <pre> Elements. 


																		    asterisk.route.intermediary.loadPage_runDefaultScript = function() {
																		        var targetedPreElements = Array.from(document.getElementsByClassName('pre-removeTabSpaces'));

																		        targetedPreElements.map(function(item) {
																		            item.innerHTML = string_remove_tabSpaces(item.innerHTML);
																		        });
																		    };

4. asterisk.route.intermediary.loadPage_onEnd()

This is the last thing that runs when loading a new Page (after the Page's content is generated, and the Page's Javascript has executed).

Example : In the current website (byandu.netlify.com) it is used to remove the faSpinner loading animation. 


																		    asterisk.route.intermediary.loadPage_onEnd = function() {
																		        asterisk.route.info.mainView.classList.remove('faSpinner', 'faSpinner-fast', 'faSpinner-lg');
																		    };

Page Links are used to load a new Pape within the View.

When a Page Link is triggered, the asterisk.route.loadPage() Function is called, the url is modified and the View is updated.

Step 1. Create an <a> Element.

Step 2. Add the __route__pageLink class.

Step 3. Add the exact path from the Root Page to the .html File, as it's href value. 


															    // Generic example 
															    <a class="__route__pageLink" href="/myHTMLFile.html"> My HTML File </a>

															    // Example from this website 
															    <a class="__route__pageLink" href="/utility/spacing.html"> Spacing </a>
															    <a class="__route__pageLink" href="/components/spoiler.html"> Spoiler </a>
															    <a class="__route__pageLink" href="/functionality/route.html"> Route (Client Side) </a>

An Event Listener is added on the document, when the asterisk-route.js JS File is parsed, which delegate the functionality for all Page Links across the App.

* It is important for SEO (crawlers) and accessibility (default functionality and accessibility information) to use <a> Elements as the Page Links.

* The Event Listener's Event Handler Function uses event.preventDefault(), to prevent navigating away from the Root Page.

In order to use the Asterisk Route locally (not on an online server), we need to define the value of the localFile_rootPage and the localFile_rootPath variables.

This must be done within both the asterisk-route.js and the asterisk-route-intermediary.js Files. The variables can be found at the top of each File.

1. localFile_rootPage : String - the file name of the Root Page

2. localFile_rootPath : String - the local path to the App Folder

* The Root Page must be directly within the App Folder, as a first-level child 


															    // Example :
															    var localFile_rootPage = 'index.html';
															    var localFile_rootPath = 'file:///c:/myWebsite/' ;

One key point in every Single-Page Application is Memory Management.

Each time a HTML Element or a JS Object is created it is stored in memory.

When they are no longer used (they have no further refferences), the Javascript Garbage Collection kicks-in and removes them from memory.

But this happens only if the Elements/Objects satisfy certain conditions - mainly that they need to not be refferenced anywhere.

Within traditional Multi-Page applications, this is handled by the fact that when you navigate to a new Page, the Document is discarded completely, and a new Document is loaded.

This clears up all the memory alocated for the now-discarded Document.

Within Single-Page Applications, we never navigate away from the Root Page, so this process never takes place.

This results in HTML Elements an JS Objects remaining in memory, even after they are no longer used, which slowly adds up and results in slowing down your app, or even crashing the browser.

When a new Page is triggered to load, the asterisk.route.loadPage() Function fires.

Within it, before loading the new content, the removeExistingElements() Function executes. This takes all existing HTML Elements from the View (from the Page which is unloaded), and removes all Parent/Child refferences.

* Otherwise they would never be Garbage Collected and a Memory Leak would occur.

If among the de-refferenced Elements there are any Asterisk Components, which were activated through {*}.initialize(), these Event Listeners are also automatically removed through {*}.checkDOM(), making them elligible for Garbage Collection. 

															    asterisk.route.intermediary.removeExistingElements = function(currentView__element) {

															        var curr_view = currentView__element || document.getElementById('__route__mainView');
															        var trashBin = document.createElement('DIV');

															        var elements = Array.from(curr_view.querySelectorAll('*'));
															        elements.map(x => trashBin.appendChild(x));
															        elements.map(function(x) { 
															            trashBin.removeChild(x);
															            x.innerHTML = '';
															            x = null;
															        });
															        trashBin.innerHTML = '';

															        for (var key in asterisk.components) {
															            if (asterisk.components.hasOwnProperty(key)) {
															                if (asterisk.components[key].checkDOM) { asterisk.components[key].checkDOM() }
															            }
															        };
															    };

An HTML Element which has any Event Listeners attached to it, is not suitable for Garbage Collection. In order to make it elligible for Garbage Collection, the Event Listeners on the Element must be removed.

But there is no available method that allows us to check if an Element has any Event Listeners attached, or which Event Listeners they might be.

This results in the developer needing to manually manage the detachment of Event Listeners. To do so it is required that:

1. The Event Listeners must be attached by refferencing a named function, as the Event Handler (in order for removeEventListener() to work)

2. The Event Listeners must be specifically removed from the Element.


															    // define the named function 
															    var myHandler = function(e){ 
															        var target = e.target;
															        // do something with the target
															    };

															    // add the Event Listener, using the named function as the Event Handler
															    myElem.addEventListener('click', myHandler); 

															    // some other code

															    // remove the Event Listener, by specifying the named function used as the Event Handler 
															    // This must be done manually 
															    myElem.removeEventListener('click', myHandler);
