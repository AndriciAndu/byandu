Client-Side Routing

Functionality : Route (Client Side)

Baseline Client-Side Routing System, for simulating SPA / AJAX-based functionality.

* Updates parts of an application, without triggering a page reload

* No hash routing required ( None of this : myDomain/#/myPage.html )

* currently supports only static paths

Normal websites/applications run on a page-by-page format, consisting on multiple HTML files. Each individual HTML page has it's own content, stylesheets and scripts. When navigating from one HTML page to another, the existing HTML page is scrapped entirely, and is replaced with a new one.

Single-Page Applications run on the premise of having only a single HTML page, which is always-loaded - and within it we define Views, which have their content modified. Navigating from one HTML page to another keeps the page intact, while only modifying the content of each View

As for every project, we must first create the Index Page.

This is the foundation of the entire application, refferencing all universally-used stylesheets/scripts, and containing the markup for the application's skeleton.

The skeleton reffers to the part of the application that remains (mostly) constant/unchanged, regardless of what page/article is loaded.

Somewhere within the skeleton, we define a div with an id="__route__mainView", which represents our View (where our page's content will actually load).

This can be compared to an iframe - we are loading a different HTML file (with extra stylesheets/scripts if desired) within our View

For example, in this website (byandu.netlify.com), the skeleton can be resumed to : 

						    <!DOCTYPE html>
						    <html>
						      <head>
						          // all the application's Meta Tags 
						          // all general-purpose CSS files 
						      </head>
						      <body>
						          <div class="body-sideNav"> ... </div>          // with all the sideNav buttons and what-nots 
						          <div class="body-main"> 
						              <div class="container"> 
						                  <div id="__route__mainView"> </div>    // this is the Main View, where each article/page is loaded 
						              </div>
						          </div>
						          <div class="modalsContainer"> ... </div>       // all modals that are used across multiple articles 
						          // all general-purpose JS files, including asterisk-route.js 
						      </body>
						    </html>

An individual page/article's HTML file is structured like so : 

						    <script src="<--myPath--/route_intermediary.js"></script>
						    <article> 
						         <!-- The Page's Content --> 
						    </article>

When navigating directly to the Root/Index Page : (ex: byandu.netlify.com )

The 'scheleton' markup is parsed and generated

The general-use CSS/JS Files are parsed and memorized

The asterisk-route.js file is parsed and a 'default' page is generated within the View

From here, navigating to a new page will simply update the View with the coresponding data.

When navigating directly to a specific Page : (ex: byandu.netlify.com/effects/appear.html )

The 'scheleton' markup is not present within the appear.html File - as it contains only the HTML for the article itself

The general-use CSS/JS Files are not refferenced

The asterisk-route.js file is not refferenced and it's functionality cannot take effect.

By itself, navigating to a specific path (without going to the Root/Index Page first), would leave the user with a basic HTML file
