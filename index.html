Different files

The 'main' (complete) files contain the code for all the available features/effects. This can be used during development, when things may change along the way.

*Spec files can be created within the code Tab, by selecting only the desired features. This is recommended for Deployment, to not include unused CSS in the finished product. (This is not required if already using an unused-CSS-removal tool)

Individual features/effects can also be used as direct code (copy-paste) or saved in individual files. It is often the case when you only need certain features or a very limited numbers of effects within your project - the code can be copy-pasted directly into your codebase or kept as a separate file for easier maintainability during Production.

There are some differences though - the 'main' files have their code more grouped together, to avoid repetition and to save on file size. It makes groups of effects easier to understand as a whole, but might make tweeking individual values a bit more demanding.

*spec files have their code mostly grouped together in the same fashion as the 'main' files, yet there is more repetition involved, ~~~~~~~~~~~~~ as they need to function individualy.

Individual codeBlocks have all their logic contained within the generated code.

Use the main files within the first phases of production - mostly until the design is definitive.

Use the spec files within the later phases of production - when the design is set, and the desired features are decided upon - changes can then be made to the files themselves.

Use the individual files/code snippets when only certain features/effects are used within the project - the most common scenario.

Utility Classes

Utility Classes are CSS Classes that serve a single, specific purpose.

* Example : The Bootstrap Grid System - <div class="col-6"> ... </div> - is formed of layout-specific Utility Classes.

An Utility Class is a chunk-of-logic which is:

* reusable : can be used across different elements

* specific : all styles are directly related to the intended style/effect

* pacifistic : doesn't create undesirable conflicts with other applied styles

* encapsulated : all logic is included / no external dependencies are required

Use Cases

They can approach a multitude of topics : .

Their usefulness is always determined by the 'coding rules' each developer/team follows.

* Which / when / where / how to use any collection of Utility Classes, along with their benefits / drawbacks can vary from project-to-project and from team-to-team.

Layouts col-6 col-12
Spacing m-0 p-xs
Alignment float-left text-center
Visibility hidden invisible
Colors btn-primary color-brand-primary
Effects hasFilter-aden boxShadow-xs
States hvr-grow focus-border-glow
and more and more and more..

Purpose

Utility classes remove the need of generating new classNames, in order to add simple styles/effects, for elements that do not hold specific semantics themselves.

If an Element can be given

Bad : Unnecessary-class
<div class="yetAnotherClassName">
.yetAnotherClassName {
text-align: center;
}
Good : Utility-class
<div class="text-center">
.text-center {
text-align: center;
}

If used on elements which don't hold specific semantics themselves, they help the separation of concerns, and in keeping the nameSpace clean.

* Another use case is in overriding certain styles/effects within already-styled Components. This use-case should be applied with consideration.
   Because of that, they are often used with the !important declaration, to override more-specific rules.

Ace up the sleeve : Responsive Utilities

One of their most important features lies within their Responsive Utilities.

This extensively helps in reducing CSS media-query-declarations repetitiveness, and allowing to easily change specific styles on different viewports.

However, not all utility classes have use for their responsive utilities in every project.

* Example: all wireframing classes are universally used, in almost every project :

<div class="col-12 sm-col-6 md-col-4">

* Example: certain textStyle classes might be changed on different viewports :

<div class="text-center md-text-left">

* Example: certain textStyle classes might almost never be changed on different viewports :

<div class="text-italic md-text-fontStyle-normal">

This is the reason why you should always use a Task Runner (such as Grunt or Gulp) to eliminate unused CSS.

You can also create a personalized build of any Asterisk Utilities, within the Code Tab, selecting only the ones that would suit your project.

Good practices recommend that they are used mainly on intermediary/container elements, which don't have a specific class themselves.

They can also be used on specific components, if the styling/effects would only slightly differ from the original.

If multiple rules are required for the change, or if the utility classes are replicated throughout the component, it would be better to simply add a class to the Element and include the styles within it.

'Dummy' Parents

Might be named differently across the internet : 'Dummy' Parents / 'Dummy' Containers / 'Utility' Parents / 'Utility' Containers / etc

Regardless the name, a 'Dummy' Parent is a simple div which, by itself, holds no specific styles/functionality - it is a generic div.

It is used to wrap a specific HTML Element and to hold certain styles/effects for the element in question.

<div> <-- this is a 'Dummy' Parent --> <div class="myElement"> ... </div> </div>

This is used to group elements or to separate concerns, regardind :

Wireframing
<div class="col-6"> <-- this is a 'Dummy' Parent --> <div class="myElem"> ... </div> </div>
Utiity Classes
<div class="text-center"> <-- this is a 'Dummy' Parent --> <div class="myElem"> ... </div> </div>
Effects
<div class="hvr-grow"> <-- this is a 'Dummy' Parent --> <div class="myElem"> ... </div> </div>
Overlays
<div class="filter-glow"> <-- this is a 'Dummy' Parent --> <div class="myElem"> ... </div> </div>

They are also used to apply certain effects, without creating conflicts with existing styles :

pseudo-elements :
apply an effect which uses pseudo-elements
on an element which already has pseudo-elements for something else
transforms :
apply an effect which uses transforms
on an element which already has transforms for something else

The 'Dummy' Parent, as a generic div, does not affect the layout or styles of myElement in any way - by itself, it's like it isn't even there.

They are added to create cleaner markup and to avoid creating conflict between effects and component styles.

As certain scenarios require additional markup, to generate a specific effect/result, elements added to the markup just to hold 'intermediary' classes/styles/roles, are ussualy reffered to as 'Dummy'/'Utility' Elements.

But, in the end, 'Dummy' Parents are divs which are added to the markup - and they should be treated like any markup changes, as they can generate bugs if for example :

* there are existing child-specific CSS rules (.something > .myElement)

* flex properties are used (which apply as child-specific CSS rules)

* existing scripts are refferencing myElement.parentElement directly

* etc - generic examples are generic

The [ asterisk ] object

The asterisk is a basic Javascript Object. It is defined in the global scope.

It holds all the functions / methods that are required for any component / effect to execute.

* It is not the most memory-efficient or highly-procutive pattern, yet it is more accessible for beginner-level vanilla-JS developers. 

										    asterisk {              // main wrapper Object 

										        components {           // wrapper Object for all Components 
										            tabs {             // wrapper Object for all of Tabs Component Functions 
										                run() { ... } ,               // Example of a Tabs Component Function 
										                ...
										            } ,
										            spoiler {          // wrapper Object for all of Spoiler Component Functions 
										                run() { ... } ,               // Example of a Spoiler Component Function 
										                ...
										            } ,
										            ...
										        } ,

										        effects {             // wrapper Object for all Effects 
										            appear {          // wrapper Object for all Appear Effect Functions 
										                checkScroll() { ... } ,  // Example of an Appear Effect Function 
										                ...
										            } ,
										            ...
										        } ,

										        utility {            // wrapper Object for all Utility Functions 
										            hasClass() { ... } ,        // Example of an Utility Function 
										            ...
										        } ,

										        route {               // wrapper Object for all Route-related Functions 
										            fetchHTML() { ... }          // Example of an Route-related Function 
										            ...
										        } ,
										        ...
										    }

It acts as an wrapper, preventing the polution of the global nameSpace, while making all functions globally-available.

It helps with avoiding memory leaks, by preventing the use of annonymous Functions as eventHandlers within eventListeners.

* It allows for the option of removing said eventListeners from the elements they are attached to, if the elements are no longer used.

Initialization

The asterisk Object is included in every asterisk-{*}.js file. It is created in the first-parsed file, and refferenced further on.

* This allows the files to be added in any order, and still function normally.

							    var asterisk = asterisk || {} ;  // refference or create the object

Main 'Categories'

The Main 'Categories' reffer to the object's 'first-level children' (properties), representing the main 'wrapper' Objects.

These are created when refferencing a asterisk-{*}.js file of a 'Category'. 

								    asterisk = {
								        components : { ... } , 
								        effects :    { ... } ,
								        ...
								    }

* Example: if using a Component which has a .js file - it will generate the asterisk.components wrapper object.

* Example : if no components are used (no component .js files are refferenced), the asterisk.components wrapper Object will not be created. 

											    // Example for the asterisk-components-tabs.js file 
											    asterisk.components = asterisk.components || {} ;     // refference or create asterisk.components wrapper object 
											    asterisk.components.tabs = {};                           // create the tabs wrapper Object

Available Methods

Each component/effect/etc has it's available Methods stored within asterisk.{category}.{item}.

If can be accessed by asterisk.{category}.{item}.{method}

Format : asterisk.{category}.{item}
Example : asterisk.components.tabs Example : asterisk.effects.appear
Format : asterisk.{category}.{item}
Example : asterisk.components.tabs.run() Example : asterisk.effects.appear.checkScroll()

Each file contains a shortcut-refference, to improve readability (and slightly reduce files size). 

							    var tabsObj = asterisk.components.tabs;  // shortcut-refference 
							    ...
							    tabsObj.run = function() { ... }

Dependencies

Every Component/Effect/etc has it's entire logic encapsulated within it's .js file.

If Utility Functions are required, they are defined at the start of the file, and are stored within the asterisk.utility Object. 

								    asterisk.utility.myFunction = asterisk.utility.myFunction || function() {
								         // function logic 
								    };

								    var myFunction = asterisk.utility.myFunction;  // shortcut-refference

This encapsulation method has its' benefits :

* Modularity : each Component/Effect can be used independently and in any order.

* Memory Efficient : Storing all Utility Functions within the asterisk Object, allows for the IIFE wrapper to be garbage-collected after the initial code runs.

But it is not without its' flaws :

* Repeating Code : each Utility Functions required has it's body/definition included within in each file, even if it was previously defined within another file.

* Mutability : even though improbable, it is possible to disrupt the functionality if any of these function are defined differently, previous to their initialization.
