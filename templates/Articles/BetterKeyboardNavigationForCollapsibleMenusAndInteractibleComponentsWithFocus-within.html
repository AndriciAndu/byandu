Collapsible/Expandable Menus or Interactible Components, which show/hide some of their children in different states, are a key part of any project.

Not done yet : tabindex="1" is a mischievous mistress

* Adding tabindex="1" attribute on each of elements is a tedious and prone-to-oversights process

* Itself, tabindex="1" cannot be modified through CSS (because it modifies behaviour - not styles)

* These elements themselves should not be declared as focusable if they don't offer direct interaction (which they don't)

Ok, ok, all warnings aside, we're done now, right?

Still not done yet : where did my :focus go ?

Now what happens if the component we are focusing, also has focusable elements within it?

What ? Why ?

2. The :focus state only takes effect when the element itself has focus

* Once the focus is moved away from the element itself (into a focusable child), the :focus on the Parent no longer applies.

* Once the focus is moved to a child element, that child element is 'scrolled-into-view', potentially offsetting the entire element.

Enough is enough : :focus-within to the rescue !

There is however a simple addition which fixes this issue: :focus-within

<div class="myMenu-1"> <div class="menuTitle"> Menu Title </div> <div class="menuItem" tabindex="1"> Menu Item </div> <div class="menuItem" tabindex="1"> Menu Item </div> <div class="menuItem" tabindex="1"> Menu Item </div> </div>
.myMenu-1 { max-height: 50px; overflow: hidden; transition: max-height .4s; } .myMenu-1:hover, .myMenu-1:focus-within { max-height: 200px; }

Adding :focus-within alongside :hover will :

  * Make the style changes take effect just like they would on :hover ;

  * Does not require making the element itself focusable - so no more need for tabindex="1"

  * Most importantly : allow keyboard users to have the same experience as those who are navigating with the mouse.

Take away:

/* Make keyboard users happy: */ .myElem:hover , .myElem:focus-within { /* do magic */ }
